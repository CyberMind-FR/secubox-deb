Architecture Approaches

How should we structure the Eye Remote Swiss Army system?

A

Unified Python Agent (Recommended)

Single Python process handles everything:

  • fb_dashboard.py — Framebuffer renderer (existing, extended)
  • FastAPI server — Web Remote at :8080 (embedded)
  • Mode state machine — Handles transitions & failover
  • System controls — WiFi/BT/Display via subprocess calls

Pros

  • Single process = simple deployment
  • Shared state, no IPC needed
  • Extends existing code
  • Low memory footprint

Cons

  • Crash affects everything
  • Blocking operations need async care
B

Microservices (Systemd Units)

Separate services communicating via Unix sockets:

  • eye-display.service — Framebuffer renderer only
  • eye-web.service — FastAPI Web Remote
  • eye-agent.service — Mode logic & SecuBox comms
  • eye-system.service — WiFi/BT/System controls

Pros

  • Isolated failures
  • Independent restarts
  • Cleaner separation

Cons

  • More complexity
  • IPC overhead
  • Higher memory on Pi Zero
C

Hybrid: Core Agent + Web Sidecar

Two processes with clear responsibilities:

  • eye-agent — Display + Mode + SecuBox (critical path)
  • eye-web — Web Remote + System controls (optional)
  • Communication via shared JSON file + signals

Pros

  • Display works even if web crashes
  • Web can be disabled to save resources
  • Simple IPC (file + SIGHUP)

Cons

  • State sync complexity
  • Two processes to manage
